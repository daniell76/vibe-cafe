# Vibe Café

In-cafe demo combining Gemini image generation with a small Next.js app for ordering, queue management, and an in-store big-screen experience. Originally built as the "Cloud in your Coffee" Google Cloud event app.

## Quick start

> **All you need is a Google Cloud project ID.** Everything else (region, service name, bucket name, AR repo, TF state bucket) has a sensible default — see [`.env.example`](./.env.example) if you want to override.

```bash
git clone <repo>
cd vibe-cafe

# 1. One-time bootstrap (enables APIs, creates Firestore DB / Artifact Registry repo /
#    TF state bucket, configures docker auth, writes terraform/backend.hcl).
#    Defaults: region=europe-west4, service=vibe-cafe.
./scripts/setup.sh YOUR_PROJECT_ID

# 2. Every deploy (docker build + push + terraform apply + prune old revisions).
./scripts/deploy.sh
```

The deploy script prints the Cloud Run URL when it's done.

Want a different region? `REGION=us-central1 ./scripts/setup.sh YOUR_PROJECT_ID` — and the deploy step will pick it up too.

**Multi-environment?** Use `gcloud config configurations` as the env switcher:
```bash
gcloud config configurations create dev   # or prod, staging, …
gcloud config configurations activate dev
gcloud config set project dev-project-id
./scripts/setup.sh dev-project-id         # once per env

# later, swap envs:
gcloud config configurations activate dev   # or prod
./scripts/deploy.sh                         # auto-discovers backend by gcloud project
```
See [`DEPLOY.md`](./DEPLOY.md) for the full multi-env workflow, including optional `.env.<project>` overrides.

## What it does

There are five user-facing surfaces, all served by one Cloud Run service:

| Route | For | What |
| --- | --- | --- |
| `/` | Customer | Four-step ordering wizard: customise → pick a foam-art preview (4 variants generated in parallel from your "happy place") → review → submit |
| `/barista` | Staff | Live order queue with **Making / Complete / Pick up** state buttons and a per-order **Download** that names the foam image `<order-num>-<customer-name>.png` for non-API foam printers |
| `/tracking` | Customer-facing wall | Continuous status display (Received / Making / Ready) |
| `/admin` | Manager | App identity, AI prompt templates (with Restore default), menu CRUD, defaults picker, analytics, ☠ Nuke-all-data |
| `/booth` | Big screen at the table | Full-screen 4K "vibe" image of the customer's happy place. Corner keypad for entering the order number; tap-to-toggle menu; in-place Regenerate button |

Two distinct images are produced per order:
- **Foam art** — 1K, 1:1, sepia-on-white stencil for foam printing
- **Vibe image** — 4K, 16:9 or 4:3 (admin-configurable), magazine-quality photography for big-screen display

Both are generated by `gemini-3.1-flash-image` on Vertex AI. The vibe gen runs as fire-and-forget background work (`next/server` `after()`) so the customer's ordering UX doesn't wait for the 4K render — by the time they reach the table, the booth has it.

## Tech

- Next.js 16 (App Router, **webpack** build — Turbopack drops styled-jsx output, see `DEPLOY.md`)
- TypeScript, React 19
- `@google/genai` for Gemini image + text generation against Vertex AI
- `@google-cloud/firestore` for orders + settings + counter
- `@google-cloud/storage` for image bytes
- Cloud Run (v2) with `cpu_idle = false` so background work runs
- Terraform for everything cloud-side

## Project layout

```
app/                  Next.js routes (UI pages + /api/...)
  api/order/preview        4-foam-previews + 1-vibe-bg endpoint
  api/order                submit; persists picked foam + vibe; deletes unpicked
  api/order/[id]/foam      download foam with Content-Disposition naming
  api/order/[id]/regenerate-vibe   re-render the vibe image for an order
  api/config + /defaults   settings get/patch + code defaults
  api/admin/nuke           wipe all orders + GCS images (admin only via UI)
components/           VibeLoader, NavBar, MenuListEditor, ordering wizard steps
lib/                  firestore.ts, storage.ts, vertex-ai.ts
terraform/            Cloud Run service + bucket + APIs
scripts/              setup, deploy, prune-revisions, prompt-test scripts
docs/design/          Source design mockups
```

## Local dev

```bash
npm install
cp .env.example .env.local
# edit .env.local → set GOOGLE_CLOUD_PROJECT
npm run dev
```

The dev server reaches the real Firestore + Vertex AI in `GOOGLE_CLOUD_PROJECT`, so you need ADC set up (`gcloud auth application-default login`).

## Deploy

```bash
./scripts/setup.sh YOUR_PROJECT_ID   # once per project
./scripts/deploy.sh                  # every deploy
```

See [`DEPLOY.md`](./DEPLOY.md) for what these scripts do, the env-var contract, and the rationale behind `cpu_idle = false`, the image proxy, and the `--webpack` build flag.

## Scripts cheat sheet

| Command | Purpose |
| --- | --- |
| `./scripts/setup.sh <project>` | One-time per-project bootstrap (APIs, Firestore, AR repo, TF state bucket, docker auth, backend.hcl) |
| `./scripts/deploy.sh` | Build, push, terraform apply, prune old revisions |
| `node scripts/prune-revisions.mjs` | Delete Cloud Run revisions beyond the latest `KEEP=3`. `DRY_RUN=1` to preview |
| `node scripts/test-foam-prompt.mjs "<happy place>"` | Compare current vs proposed foam prompt locally |
| `node scripts/test-vibe-prompt.mjs ["<happy place>"]` | Compare current vs proposed vibe prompt locally (4K) |
| `npm test` | Vitest |
| `npm run lint` | ESLint |
| `npm run build` | Production build (`next build --webpack`) |

## Tests

`tests/` covers the storage, firestore, vertex-ai mocking, and the order API contract. Run `npm test`.
